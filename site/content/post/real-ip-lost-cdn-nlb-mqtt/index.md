> **核心结论（TL;DR）：**
>
> 在 `Client → CDN → NLB(TCPTLS) → EMQX` 这条链路中，由于 CDN 终结了客户端的原始 TCP 连接并重新建连到 NLB，**真实客户端 IP 在 CDN 这一跳就断了传递链**。而 NLB 做 TLS 卸载时无法兼容 Proxy Protocol 的接力转发，XFF 又因为 MQTT 不是 HTTP 协议而无法使用，最终形成了一个"不可能三角"的工程困局。

<!--more-->

## 一、 问题背景：后端日志里全是 CDN 的 IP

我们的业务是 IoT 设备通过 MQTT 协议连接云端的 EMQX 消息服务器，整条链路如下：

```text
IoT 设备 --MQTTS(TLS)--> 网宿CDN(L4加速) --TCP--> 阿里云NLB(TCPTLS卸载) --> EMQX
```

某天，安全团队要求我们在 EMQX 侧记录每个设备连接的**真实客户端 IP**，用于审计和异常检测。结果打开日志一看——**所有连接的源 IP 都是网宿 CDN 节点的 IP**，没有一个是设备的真实 IP。

这很好理解：CDN 作为中间代理，终结了设备的原始 TCP 连接，然后用自己的 IP 重新建连到 NLB。NLB 和 EMQX 自然只能看到 CDN 的 IP。

**但问题是：怎么把真实 IP 传过来？**

---

## 二、 科普：传递真实 IP 的三种主流方案

在动手之前，先搞清楚业界传递真实客户端 IP 的三种主流技术，它们各自工作在不同的协议层：

| 技术 | 工作层级 | 原理 | 局限 |
|------|---------|------|------|
| **XFF** (X-Forwarded-For) | L7 HTTP 头 | 代理在 HTTP 请求头里追加 `X-Forwarded-For: <客户端IP>` | **只适用于 HTTP 协议** |
| **Proxy Protocol (PP)** | L4 TCP 连接层 | 在 TCP 连接建立后、应用数据前插入一行文本：`PROXY TCP4 <src> <dst> <port> <port>\r\n` | 链路中每个节点都必须支持解析 |
| **TOA** (TCP Option Address) | L4 TCP 选项字段 | 把客户端 IP 塞进 TCP 头的 Option 字段（通常 kind=254） | 需要服务端装内核模块，且中间设备不能剥离 TCP Option |

### Proxy Protocol 到底长什么样？

很多人知道 PP 但没见过它的真面目。它其实就是 TCP 连接建立后、应用数据之前的**一行纯文本**：

```text
[TCP 三次握手完成]
PROXY TCP4 192.168.1.100 10.0.0.1 56789 1883\r\n   ← 这就是 PP 头
\x10\x3f\x00\x04MQTT...                              ← 真正的 MQTT CONNECT 报文
```

接收方（比如 EMQX）读到第一行 `PROXY...`，解析出真实客户端 IP 后剥掉这行，然后把后面的数据当成正常的 MQTT 协议处理。

**关键认知：PP 头不是 HTTP 头，也不是 TCP Option，它是夹在 TCP 连接和应用层之间的一块"夹心饼干"。** 链路中任何一个节点如果不认识它，就会把它当成脏数据，导致上层协议握手失败。

## 三、 排查过程：逐个方案的尝试与失败

### 方案 1：XFF —— 第一个被排除的

XFF 是 HTTP 专属的请求头，我们的协议是 MQTT over TCP，不存在 HTTP 头这个概念。

**结论：❌ 协议不兼容，直接排除。**

### 方案 2：Proxy Protocol —— 最有希望，却最痛

PP 在 L4 层工作，不依赖 HTTP，理论上最适合 MQTT 场景。而且 EMQX 原生支持 PP（配置 `proxy_protocol = on` 即可）。

于是我们让网宿 CDN 开启 PP，同时阿里云 NLB 也开启了 PP。

**结果：所有 MQTT 连接全部失败。**

#### 为什么？—— 双层 PP 冲突

联系阿里云技术支持后，得到了答案：

```text
EMQX 收到的数据流：

PROXY TCP4 <CDN的IP> <NLB的IP> ...\r\n        ← ① NLB 自己加的 PP 头
PROXY TCP4 <真实客户端IP> <CDN的IP> ...\r\n   ← ② 网宿 CDN 加的 PP 头（被当成数据透传了）
\x10\x3f\x00\x04MQTT...                       ← ③ 真正的 MQTT 数据
```

NLB 在监听端确实解析了 CDN 发来的 PP 头，但在转发给 EMQX 时，它做了两件事：
1. 加了一个**自己的** PP 头（① 里的内容）
2. 把 CDN 原始的 PP 头文本**当成数据**一起透传了下去（②）

EMQX 解析完第一个 PP 头后，期望的下一个字节是 MQTT CONNECT 报文（`0x10`），结果迎面撞上第二个 `PROXY TCP4...` 文本——**协议解析直接炸了**。

> 💡 **问题本质**：阿里云 NLB 没有实现 **PP 接力（PP Relay）** 能力——即"解析上游 PP → 剥离原始 PP → 提取真实 IP → 用该 IP 生成新的单层 PP 转发给后端"。相比之下，HAProxy 的 PP 实现是业界标杆，能完美处理这种多层代理场景。

#### 那 CDN 不开 PP，只让 NLB 开呢？

试过。NLB 确实能加 PP 头给 EMQX，但它写入的源 IP 是 **TCP 连接的直接上游**——也就是 CDN 的 IP，不是客户端的 IP。因为 CDN 没有通过任何方式把真实 IP 传递给 NLB，NLB 根本不知道真实客户端是谁。

**结论：❌ 单开 NLB 的 PP 拿到的是 CDN 的 IP；双开 PP 则数据流被污染。**

### 方案 3：TOA —— 最后的希望，也破灭了

#### TOA 是怎么工作的？

TOA 的思路非常巧妙：它把客户端的真实 IP **藏在 TCP 协议头的 Option 字段里**。

TCP 头部有一个可变长的 Option 区域（最大 40 字节），通常用于承载 MSS、窗口缩放、时间戳等信息。TOA 就是在这里"夹带私货"——代理服务器在 TCP 三次握手的 SYN 包中，往 Option 里塞入一个自定义字段（通常 kind=254），内容是客户端的真实 IP 和端口：

```text
TCP Header
├── Source Port: 54321
├── Dest Port: 1883
├── Seq / Ack / Flags ...
└── Options:
    ├── MSS: 1460
    ├── Window Scale: 7
    ├── Timestamps: ...
    └── TOA (kind=254): 203.0.113.50:8823   ← 真实客户端 IP 藏在这里！
```

后端服务器装上 TOA 内核模块后，应用程序调用 `getpeername()` 获取对端 IP 时，内核模块会"偷梁换柱"——不返回直连上游的 IP，而是返回 TCP Option 里藏的真实客户端 IP。**对应用层来说完全透明，不需要改一行代码。**

#### 为什么在我们的场景下也不行？

问题出在 NLB 这一跳。画出来就明白了：

```text
CDN 到 NLB 的连接（TCP 连接 A）：
┌─────────────────────────────────────────────────┐
│ SYN 包的 TCP Option 里带着 TOA: <真实客户端IP>     │
│ 源 IP: CDN    目的 IP: NLB                        │
└──────────────────────┬──────────────────────────┘
                       ↓
              NLB 收到 SYN，终结 TLS
              TCP 连接 A 在此终结 ❌
                       ↓
              NLB 重新建立到 EMQX 的连接
                       ↓
NLB 到 EMQX 的连接（TCP 连接 B）：
┌─────────────────────────────────────────────────┐
│ SYN 包的 TCP Option：MSS, Window Scale, TS...     │
│ ❌ 没有 TOA！这是一条全新的连接！                    │
│ 源 IP: NLB    目的 IP: EMQX                       │
└─────────────────────────────────────────────────┘
```

**NLB 做 TLS 卸载时，会终结上游（CDN）的 TCP 连接，然后创建一条全新的 TCP 连接到后端（EMQX）。** 这条新连接的 SYN 包是 NLB 自己构建的，它不会把上游连接的 TCP Option 照搬过来——**TOA 信息就在这一跳彻底丢失了。**

这和 PP 的双层冲突是不同的失败原因，但结局一样：真实 IP 传不到 EMQX。

**结论：❌ NLB 终结连接后重新建连，TCP Option（TOA）不会被携带到新连接上。**

> 💡 **延伸：TOA 到底什么时候能用？**
>
> TOA 的核心前提是：**中间设备不能终结 TCP 连接**。只要某一跳"断开旧连接 + 新建连接"，TCP Option 就会丢失。TLS 终结只是导致连接被断开重建的最常见原因之一：
>
> | 中间设备行为 | 是否断开重建连接 | TOA 能否存活 |
> |-------------|----------------|-------------|
> | LVS DR 模式（直接路由） | ❌ 不断开，改 MAC 转发 | ✅ 能 |
> | LVS NAT 模式 | ❌ 不断开，改 IP 转发 | ✅ 能 |
> | NLB/CLB TLS 卸载 | ✅ 断开重建 | ❌ 丢失 |
> | HAProxy / Nginx（L4/L7 代理） | ✅ 断开重建 | ❌ 丢失 |
>
> 所以 TOA 真正适用的场景其实很窄——**只有纯转发（不终结连接）的四层负载均衡才能透传 TCP Option**。这也是为什么 TOA 主要在云厂商的 L4 LB（LVS DR/NAT 模式）里用得好，一旦链路里出现任何代理或 TLS 终结行为就会失效。

### 方案 4 & 5：能工作但不可接受的方案

| 方案 | 改动 | 为什么不行 |
|------|------|-----------|
| NLB 改纯 TCP 模式，不做 TLS 卸载 | CDN 开 PP → NLB 透传 → EMQX 解析 PP ✅ | EMQX 自己扛 TLS 握手，海量 IoT 设备连接下 CPU 开销巨大，**EMQX 资源消耗不可接受** |
| NLB 换成 CLB + 自建 HAProxy | HAProxy 完美接力 PP ✅ | 链路多了一层自建组件，运维复杂度飙升，**架构过于笨重** |

## 四、 本质分析：一个"不可能三角"

把所有约束画在一起，会发现这是一个经典的工程困局：

```text
          获取真实客户端 IP
                🔺
               /  \
              /    \
             /      \
        NLB 做       链路简洁
       TLS 卸载      不加组件
```

- **要真实 IP + TLS 卸载** → 需要 HAProxy 做 PP 接力 → 链路笨重 ❌
- **要真实 IP + 链路简洁** → NLB 纯 TCP 透传 PP → EMQX 自己扛 TLS ❌
- **要 TLS 卸载 + 链路简洁** → NLB 做 TCPTLS，但拿不到真实 IP ❌

**三个目标，最多只能同时满足两个。**

## 五、 反思与建议

### 这个锅应该谁背？

根本原因是**阿里云 NLB 的 Proxy Protocol 实现不完整**。一个成熟的 L4 负载均衡器应该支持 PP 接力：

```text
理想的 NLB PP 接力行为：

CDN → NLB:  PROXY TCP4 <真实IP> <CDN_IP> ...\r\n + 应用数据
               ↓ NLB 解析并剥离上游 PP
               ↓ 提取真实 IP
               ↓ 生成新的单层 PP 转发
NLB → EMQX: PROXY TCP4 <真实IP> <NLB_IP> ...\r\n + 应用数据
```

HAProxy 就是这么做的。但阿里云 NLB 显然没有实现这个剥离逻辑，导致上游 PP 头泄漏到数据流中。

### 更深一层：为什么 NLB 做不到而 HAProxy 可以？

这和它们的底层架构基因有关。阿里云 NLB 的底座是 **LVS（Linux Virtual Server）**，而 HAProxy 是完全不同的物种：

**LVS** 工作在**内核空间**，本质是一个**包级别（packet-level）的转发器**。数据包进来，改改 IP/端口，直接转发出去。它看到的是一个一个的 IP 数据包，不理解"TCP 连接"和"数据流"的概念。快得飞起（内核态 + DPDK，百万级 CPS），但也因此**看不懂 TCP 字节流里藏的 PP 头**。

**HAProxy** 工作在**用户空间**，本质是一个**连接级别（connection-level）的代理**。它终结上游连接、解析完整 TCP 流、理解 PP 头、剥离、建新连接、加新 PP 头、转发——这些流级别操作对它来说都是基本功。更何况，**Proxy Protocol 就是 HAProxy 的作者 Willy Tarreau 发明的**，自然支持得最完美。

| 特性 | LVS（NLB 底座） | HAProxy |
|------|----------------|---------|
| 工作空间 | 内核态 | 用户态 |
| 处理粒度 | IP 数据包 | TCP 连接/流 |
| 性能 | 极高（百万级 CPS） | 高（十万级 CPS） |
| 能否理解 PP 头 | ❌ 天然不理解字节流内容 | ✅ 原生支持（PP 的发明者） |
| PP 接力能力 | ❌ | ✅ |

所以 NLB 的 PP 支持大概率是在 LVS 之上**打了补丁**实现的——能做到"往后端连接前面加一行 PP 头"（这相对简单，只是在转发时 prepend 一段字节），但**"从上游读取并剥离 PP 头再接力"这种需要解析 TCP 字节流的操作，LVS 的包转发架构天然不擅长**，所以没做完整。

> 💡 **一句话总结：LVS 是"搬运工"，只管搬箱子不看里面装了什么；HAProxy 是"翻译官"，会拆开信封读内容再重新封装。PP 接力需要的恰恰是"翻译官"的能力。**

### 给大家的建议

1. **在架构设计阶段就考虑真实 IP 传递问题**。不要等到安全审计时才发现——等链路固化了再改，代价极大。
2. **多层代理 + 非 HTTP 协议 = 地狱级难度**。HTTP 有 XFF 这个成熟方案，但 MQTT、gRPC 等 TCP 协议在多层代理下传递真实 IP 几乎没有银弹。
3. **不要假设"开启 PP"就万事大吉**。PP 的核心痛点不在于"开不开"，而在于**每一跳的处理逻辑是否正确**——是剥离接力，还是叠加透传？这决定了成败。
4. **选型时确认 LB 的 PP 接力能力**。如果你的链路超过两层代理，请在选型阶段就验证 LB 是否支持解析上游 PP 并接力转发，而不是在生产环境踩坑。

## 结语

技术世界里，最无奈的不是"做不到"，而是"每个方案都差一点就能行"。这次排查让我深刻体会到：**理论上完美的协议设计（Proxy Protocol），在真实的多层架构中可能会因为某一个组件的实现缺陷而彻底失效。**

如果你也遇到了类似的困局，希望这篇文章能帮你少走一些弯路。也期待阿里云能在未来的 NLB 版本中补齐 PP 接力的能力——这对所有非 HTTP 协议的用户来说，真的太重要了。
