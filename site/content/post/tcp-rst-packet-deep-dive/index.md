> **核心结论（TL;DR）：**
>
> 当你在日志里看到 `Connection reset by peer` 时，意味着对方（peer）向你发送了一个 TCP **RST（Reset）** 包，强行中断了连接。RST 就像是 TCP 世界里的"掀桌子"——不打招呼、不走流程、直接翻脸。

<!--more-->

## 一、 什么是 RST 包？（为什么要掀桌子？）

我们知道，正常的 TCP 断开连接需要通过**四次挥手**（FIN → ACK → FIN → ACK），这是一种温柔的、有商有量的告别方式。

但 RST 完全不一样。它是 TCP 协议头里一个特殊的标志位（Reset flag），一旦被置为 1，就代表发送方在大喊：**"这条连接作废！不用挥手了！立刻、马上、滚！"**

收到 RST 的一方会：

1. **立刻释放**这条连接占用的所有资源（内存、端口）。
2. **不再回复**任何数据包（连 ACK 都不发）。
3. 向上层应用报一个大大的错误：`Connection reset by peer`。

> 💡 **补充**：RST 和 FIN 的关键区别在于——FIN 是"我说完了，等你确认"，RST 是"我不管你说没说完，这条连接现在就死了"。此外，向一个已经关闭的 socket 写入数据，也会触发对端回复 RST。

## 二、 谁在"掀桌子"？（三大真凶排查）

知道了 RST 的含义后，真正的难题是：**到底是谁在发 RST？为什么要发？**

以下是生产环境中最常见的三大真凶：

### 真凶 1：对方进程崩了（最直接）

- **场景还原**：你的 Nginx 日志突然报了一行 `Connection reset by peer`，而它的上游是一台 Tomcat 服务器。
- **底层原理**：Tomcat 进程因为 OOM（内存溢出）被操作系统强制杀掉了。进程死了之后，它打开的所有 Socket 文件都会被**操作系统内核回收**。此时，内核面对这些半开着的连接，不会温柔地走四次挥手（进程都没了，谁来挥？），而是对每个连接的对端**暴力发送一个 RST 包**。
- **排查手段**：直奔被告（peer）所在的服务器，看系统日志（`dmesg` / `journalctl` / `oom-kill`）或者应用自身的错误日志，看看那一瞬间是不是进程重启了或崩溃了。

### 真凶 2：中间的"第三者"强行掐断（最隐蔽）

- **场景还原**：你的请求链路是：`客户端 -> 阿里云 SLB / WAF / 防火墙 -> 你的服务器`。客户端发了一个请求，但你的服务器查数据库查了整整 2 分钟都没回包。
- **底层原理**：阿里云 SLB 或中间的防火墙通常有一个"空闲连接超时时间"（比如 60 秒）。它发现你们俩 60 秒都没说话，觉得占着茅坑不拉屎，于是**中间件伪装成服务端**，向你的客户端发了一个 `RST` 包，强行把连接掐断。
- **排查手段**：排查链路中间到底经过了哪些网关/负载均衡设备，检查它们的**超时时间（Timeout）** 配置。通常把中间件的超时时间改大，或者让程序发心跳包就能解决。

### 真凶 3：全连接队列被打满（高并发专属）

- **场景还原**：又是大促，流量如海啸。你的 Nginx 把请求疯狂打给后端的 Tomcat。
- **底层原理**：Tomcat 处理不过来了，内核的**全连接队列（Accept Queue）** 满了。此时如果内核参数 `net.ipv4.tcp_abort_on_overflow = 1`，那么面对新涌进来的连接请求，Linux 内核不仅不排队，还会直接暴躁地回一个 `RST` 包，告诉你"老子不干了，滚！"
- **排查手段**：使用 `ss -lnt` 命令，看看后端的 `Send-Q`（队列容量）是不是太小了。通过调大内核参数 `somaxconn` 和应用的 `backlog` 来扩容。

> ⚠️ **注意**：`tcp_abort_on_overflow` 的**默认值为 `0`**，即队列满时静默丢弃 SYN 包（不发 RST），客户端会等待超时后重试。设为 `1` 后才会主动发 RST 快速失败，这在排查问题时反而更友好——至少你能在日志里看到明确的报错，而不是莫名其妙的超时。

## 三、 运维老鸟的"破案 SOP"（请直接背诵）

以后再看到 `Connection reset by peer`，不要慌，按照这三步走：

1. **定界（确定 peer 是谁）**：
   - 仔细看架构图。这个报错是发生在哪两层之间的？是 `浏览器 -> Nginx`，还是 `Nginx -> Tomcat`，还是 `Tomcat -> MySQL/Redis`？明确谁是被告（peer）。

2. **查日志（看有没有人死掉）**：
   - 直奔被告（peer）所在的服务器，看那一瞬间应用有没有重启？系统日志（`dmesg` / `/var/log/messages`）有没有报 OOM？

3. **抓包（终极杀器 tcpdump 出场）**：
   - 如果谁都没死，大概率是中间件捣鬼或者协议异常。这时候就必须在服务端用 `tcpdump` 抓包了。
   - **命令示例**：

   ```bash
   tcpdump -i eth0 tcp port 8080 and 'tcp[tcpflags] & tcp-rst != 0' -nn
   ```

   *(这句神仙命令的意思是：只抓 8080 端口上发生的 RST 掀桌子包！极大地过滤了噪音。)*
