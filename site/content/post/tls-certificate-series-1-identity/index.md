当我们在浏览器地址栏输入一个网址，敲下回车，看到那个小小的"安全锁"亮起时，一段加密的 HTTPS 隧道就已经在眨眼间建立完毕了。

很多人都知道这把锁代表着"安全"和"加密"，但在这条隧道的基石——**TLS 证书**的底层，客户端与服务器其实经历了一场极其精妙的试探与博弈。从今天开始，我们将用四篇文章，彻底拆解 TLS 证书背后的硬核机制。

第一篇，我们先不聊高深的密码学，只聊两个最基础、但也最容易踩坑的问题：**如何证明"你是谁"，以及在茫茫网络中如何知道"连给谁"。**

## 1. 时代的眼泪：CN 的没落与 SAN 的全面上位

在配置 HTTPS 时，很多新手（甚至是一些老兵）都踩过这样一个巨坑：**明明自己签发的证书没过期，域名也填对了，但 Chrome 浏览器就是无情地划上一道红线，提示"您的连接不是私密连接"。**

这个问题，90% 是因为你还在使用已经被时代抛弃的 **CN（Common Name）**。

### 曾经的绝对主角：Common Name
在早期的 X.509 证书标准中，证书的拥有者信息里有一个字段叫 `Subject`，里面包含了国家（C）、组织（O）等信息。其中最重要的就是 **CN（Common Name，通用名称）**。

在很长一段时间里，浏览器就是靠比对你访问的域名和证书 CN 字段里的值，来确认"你是谁"的。比如你访问 `google.com`，浏览器就会去看证书的 CN 是不是 `google.com`。由于它的结构限制，通常只能存一个域名，导致运维人员必须为每一个域名单独申请一张证书，管理起来简直是噩梦。

### 破局者：万能身份容器 SAN (Subject Alternative Name)
为了解决这个问题，X.509v3 标准引入了一个强大的扩展字段：**SAN（主题备用名称）**。

很多人以为 SAN 只是一个简单的"多域名列表"，但实际上，**它是一个极其强大的"万能身份容器"**。在底层配置时，SAN 支持多种硬核的身份类型：

* **绝对主力：DNS Name (`dNSName`)**
    这是大家最熟悉的类型。无论是顶级域名 `example.com`、子域名 `api.example.com`，还是泛域名 `*.example.com`，全部都归属在这里。现代浏览器默认就是在这个字段里检索域名。
* **硬核玩家：IP Address (`iPAddress`)**
    没错，**TLS 证书是可以直接绑定给 IP 地址的！** 不仅支持 IPv4，也支持 IPv6。
    * **避坑指南：** 如果你要给 `1.1.1.1` 配置 HTTPS，在生成证书时，绝不能把它当成普通字符串塞进 DNS Name 里，必须明确指定类型为 `iPAddress`！另外，公有 CA（如 Let's Encrypt）只会给拥有绝对控制权的**公网 IP** 签发证书，内网 IP（如 `192.168.x.x`）只能靠企业自建私有 CA 解决。
* **其他进阶类型：** 比如用于企业级邮件加密的 Email Address (`rfc822Name`)，以及在云原生微服务架构（Service Mesh/SPIFFE）中大放异彩的 URI (`uniformResourceIdentifier`)。

### 权力的交接：Chrome 的绝杀
其实早在 2000 年的标准（RFC 2818）中，就已经明确规定：**如果有 SAN，就必须用 SAN，忽略 CN。** 但各大浏览器为了兼容性一直"网开一面"。

直到 2017 年，**Google Chrome 58 版本祭出绝杀：彻底移除对 CN 的后备支持。**

这意味着：**只要你的证书里没有配置 SAN 扩展字段，哪怕你的 CN 写得再对，这张证书也会被现代浏览器直接判定为无效。** CN 如今彻底沦为了一个"仅供人类阅读"的展示字段。下次用 OpenSSL 自签证书，千万别忘了配置 `subjectAltName` 扩展！

## 2. SNI (Server Name Indication)：解决"一房多客"的尴尬

解决了"证书里写谁"的问题，我们再来看看握手时的历史级难题：**单 IP 托管多域名的尴尬。**

在早期的 HTTPS 时代，运维界有一个极其硬核且昂贵的限制：**一个 IP 地址只能绑定一个 TLS 证书。**

### 经典的"先有鸡还是先有蛋"
这涉及到了 TLS 的握手顺序：
1.  **TCP 建立：** 客户端通过 IP 地址连接服务器。
2.  **TLS 握手开启：** 客户端发送 `ClientHello`，服务器立刻下发 TLS 证书。
3.  **HTTP 请求：** 只有在加密隧道建好后，客户端才发送包含 `Host` 头部的 HTTP 请求。

**死锁出现了：**
假设一台服务器（1 个 IP）上托管了淘宝和天猫。当客户端刚连上 IP 时，加密隧道还没建好，**服务器根本还没收到 HTTP 请求，不知道客户端想访问谁。** 既然不知道，服务器怎么下发正确的证书？发错了，浏览器当场报红。

因为这个死锁，当年的 HTTPS 网站必须每个都分配一个独立的公网 IP，极其浪费资源。

### 救世主：SNI 扩展
为了打破僵局，TLS 协议引入了 **SNI（Server Name Indication，服务器名称指示）**。

原理直击痛点：**既然不能在加密后告诉服务器域名，那就在加密前的 TLS 握手第一步提前说！**

有了 SNI 之后，客户端在发送 `ClientHello` 消息时，会在里面附带一个 `server_name` 扩展字段，明文写上：`tmall.com`。服务器收到后，精准地抽出天猫的证书下发。**SNI 的出现，拯救了整个虚拟主机生态**，让成千上万个 HTTPS 网站能够共享同一个公网 IP。

---

## 结语

通过 **SAN** 扩展，证书拥有了容纳多身份（域名、IP）的灵活性；通过 **SNI** 机制，服务器拥有了单 IP 托管海量域名的能力。它们共同锚定了 HTTPS 建立前最核心的身份认证基础。

但是，当服务器把证书发给你之后，浏览器凭什么相信这张证书不是黑客伪造的呢？信任的源头究竟来自哪里？
