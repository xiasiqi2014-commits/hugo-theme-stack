> **核心结论（TL;DR）：**
>
> 很多运维遇到高并发丢包，第一反应是去改 `ulimit`，结果发现请求依然大面积超时。为什么？
>
> 因为在 Linux 中"一切皆文件"，每一个建立成功的网络连接（Socket）都会消耗一个应用层的文件句柄。但这仅仅是限制并发的**"第二道门"**。 如果你不修改底层内核参数（如 `nf_conntrack`），当海量流量涌入时，**"第一道门"（内核网络栈）**就已经决绝地拒绝接客了。内核根本不让你建立连接，你应用层准备再多的文件句柄也是白搭！

<!--more-->

## 一、 案发现场：一次诡异的丢包排查

大促期间，并发流量突增。为了抗住高并发，作为运维的我习惯性地打开了服务器，一顿操作把 `ulimit -n` 调到了 100 万，满心欢喜地以为稳了。

结果没过多久监控报警，前端依然大面积请求超时！更诡异的是现象完全对不上号：

1. **查应用日志（Nginx/Java）**：干干净净，没有任何报错，甚至连访问记录都没有。

2. **查系统级日志（dmesg / /var/log/messages）**：满屏极其刺眼的红字：

   `nf_conntrack: table full, dropping packet`

明明我已经把最大连接数改到了 100 万，到底是谁在底层偷偷把我的网络包丢了？

## 二、 深入内核：阻挡并发的"两道门"

要解释这个诡异的现象，我们必须抛弃"改一个参数保平安"的单点思维，重新认识 Linux 处理网络请求的完整链路。我把它总结为阻挡并发的"两道门"。

### 第一道门：小区大门（内核层的 Netfilter 登记簿）

- **核心机制**：数据包刚进网卡，还没交到应用程序手里时，底层的连接跟踪模块（`conntrack`）会像保安一样，在一个"登记簿"上记录每一条连接的状态。这个登记簿的容量，是由 `net.netfilter.nf_conntrack_max` 决定的。
- **致命现象**：如果这道门没扩容（现代系统默认往往只有 6 万多），当第 65537 个用户的请求来到网卡时，登记簿满了。**内核直接把这个新建连接的数据包（SYN）丢弃（drop）！**
- **结果**：客户端苦苦等待直到超时。由于包死在了底层，Nginx 连包的影子都见不到，自然没有任何应用层报错。

### 第二道门：家门口（应用层的文件句柄）

- **核心机制**：过了第一道门的数据包，终于来到应用进程面前。进程要接待它，就必须向操作系统申请创建一个 Socket 文件。这就受限于 `ulimit -n`（单进程天花板）和 `fs.file-max`（系统总闸）。
- **致命现象**：如果第一道门过了，但你没改 `ulimit`，当并发超过 1024 时，应用日志就会疯狂报错 `Too many open files`。

**灾难重演：**

如果只改 `ulimit` 不改 `conntrack`，就像是你家里准备了 100 万双拖鞋，但小区大门的保安只放 6 万人进来，剩下的全堵在马路上（底层静默丢包）。

## 三、 老鸟绝对会防一手的"三大隐藏盲区"

如果你以为同时改了 `conntrack` 和 `ulimit` 就万事大吉了，那接下来的三个盲区随时可能让你的服务再次崩溃。

### 盲区一：敲了 ulimit 命令，为什么应用还是报错？（Systemd 陷阱）

很多同学在终端敲了一句 `ulimit -n 1000000`，然后去 `systemctl restart nginx`，结果发现不生效！

- **真相**：你在终端敲的命令只对当前的 bash 及其子进程生效。而现代 Linux 的后台服务全是 `systemd`（1 号进程）托管的，它根本不继承你的环境变量，甚至**无视 `/etc/security/limits.conf`**！
- **解法**：必须修改服务自己的 unit 文件。执行 `systemctl edit nginx`，在配置里写入 `LimitNOFILE=1000000`，然后 `systemctl daemon-reload` 才算真正生效。

### 盲区二：强行放大 conntrack，搞垮了 CPU？（哈希表陷阱）

内核是用"哈希表"来查这 100 万条记录的。如果你只放大了总记录数（`max`），却没放大哈希表的目录大小（`hashsize`），会导致严重的哈希冲突。

- **真相**：每次来一个包，内核要在极长的链表里挨个比对，导致 CPU 的软中断（`si`）瞬间飙升，服务器直接卡死。
- **解法**：改了 `max`，必须配套修改 `hashsize`（通常建议是 max 值的 1/4）。

### 盲区三：光顾着"接客"，忘了"做客"（临时端口耗尽陷阱）

你的 Nginx 通常是个反向代理，它接到用户的请求后，还要作为**客户端**去调用后端的 Java/Go 接口。

- **真相**：Linux 作为客户端发起连接时，需要消耗本地的临时端口。如果并发极大，且存在大量 `TIME_WAIT` 状态，临时端口（默认两万多个）瞬间耗尽，Nginx 向后端转发请求会全部失败，报错 `Cannot assign requested address`。

## 四、 终极解法：高并发调优的"系统工程"抄作业清单

真正的高并发调优，缺一不可。请把以下配置作为生产环境的标准基线：

**1. 打开第一道门（修改 `/etc/sysctl.conf`）**

```ini
# 扩大内核连接追踪表的容量
net.netfilter.nf_conntrack_max = 1048576
# 缩短 TCP 已建立连接的超时时间，加速回收
net.netfilter.nf_conntrack_tcp_timeout_established = 300
# 扩大本地临时端口范围，防止反向代理端口耗尽
net.ipv4.ip_local_port_range = 1024 65535
```

*(注意：还需要在 `/etc/modprobe.d/` 下配置 `options nf_conntrack hashsize=262144` 扩大哈希表)*

**2. 打开第二道门（系统总闸与进程限制）**

```ini
# /etc/sysctl.conf (系统总闸)
fs.file-max = 1048576
fs.nr_open = 1048576

# /etc/security/limits.conf (单用户/进程软硬限制)
* soft nofile 1000000
* hard nofile 1000000
```

*(别忘了 Systemd 托管的程序需要单独配置 `LimitNOFILE`)*
